Scope: post-LBDC-launch regression check, SFBL safety, multi-tenant
isolation, security model of the new passwordless flows, date/time
correctness, and code hygiene around the ~50 commits in the last 48

Approach: code-level review of every file in the worktree's `app/`,
`components/`, `lib/`, `scripts/`. Could not execute live Firestore

## 1 — Critical (will break for someone, fix today)

### C1. SFBL roster pages render empty — `where("status", "==", "active")` filter excludes every SFBL player doc

`/players` and `/teams/[teamId]` both query with
`.where("status", "==", "active")` to drop LBDC orphan docs. That
filter is a Firestore equality — it skips docs missing the field
entirely. The provision script (`scripts/provision.ts:393-405`) writes
SFBL players with `active: true` but **no `status` field**, so every
single SFBL player doc fails the filter. End result: SFBL `/players`
shows zero rows; every SFBL team page shows an empty roster. This is
the most-trafficked public page set on SFBL.

Fix path: either (a) drop the `status == "active"` filter in favor of
the in-memory predicate used elsewhere (which accepts undefined
status) or (b) backfill `status: "active"` on every SFBL player doc.
Option (b) needs a one-shot script that walks `/leagues/sfbl/players`
and sets the field; otherwise every newly-provisioned tenant will
hit the same wall.

- `app/players/page.tsx:73-76` — server query with the filter
- `app/teams/[teamId]/page.tsx:96-99` — same pattern
- `scripts/provision.ts:402` — writes `active: true` only, no status
- The forgiving predicate (correct shape) lives in
  `app/captain/page.tsx:138-143`, `app/captain/lineup/page.tsx:96-102`,
  `app/captain/box-score/page.tsx:2432-2434`,
  `components/captain/AttendanceTab.tsx:118-119`,
  `app/api/team-roster/route.ts:69-74` and others — use that instead.

### C2. LBDC pregame-reminder cron silently never fires

`app/api/pregame-reminder/route.ts:111` computes
`startMs = new Date(String(game.date)).getTime()` and checks whether
that falls inside `[now+45min, now+75min]`. For LBDC, `game.date` is
"YYYY-MM-DD" only and the start hour lives in a separate `game.time`
field. `new Date("2026-05-16")` parses as 00:00:00 UTC = 8 PM EDT
the day before, so a 12 PM Saturday game's `startMs` is 16 hours off
and never crosses the window. Captains receive no
"⚾ Game in 1 hour" pushes for LBDC games.

Fix: combine date+time with the existing `combineDateTime()` helper
in `lib/format-time.ts:19-38` before constructing the Date. Same
pattern the Ticker already uses (`lib/site-data.ts:116`).

### C3. iCal feed `/api/schedule.ics` produces wrong start times for LBDC

`app/api/schedule.ics/route.ts:74` calls `new Date(date)` on the raw
`game.date` field — same shape mismatch as C2. Calendar subscribers
(Apple, Google) get every LBDC game stamped as midnight-UTC =
8 PM EDT the previous day. The README banner on `/captain#schedule`
("Subscribe to your team's schedule") is the entire point of the
ICS feed, and it's currently shipping wrong data.

Same fix: stitch `data.date` + `data.time` via `combineDateTime()`
before `new Date(...)`. Bonus: line 90's `DTSTAMP:${formatICalDate(new Date())}` is fine,
but every per-event `DTSTART/DTEND` is built from the bogus Date.

### C4. SFBL captain box-score editor missing R and SB columns

`lib/tenants.ts:201` hardcodes SFBL's `stat_columns:
["ab","h","2b","3b","hr","rbi","bb","so","avg"]`. `filterCols()` in
`app/captain/box-score/page.tsx:179-195` selects from `ALL_COLS`
intersected with the config list. There is no `"r"` and no `"sb"`
entry, and `"avg"` doesn't exist in ALL_COLS at all (it's a derived
stat). So SFBL captains can't enter Runs scored or Stolen Bases —
the two most common counting stats after AB/H. Stats recalc
(`lib/stats/index.ts`) computes R and SB from box-score lineup, so
zero entry = wrong league leaders.

Fix: amend the SFBL hardcoded config to
`["ab","r","h","doubles","triples","hr","rbi","bb","so","sb"]`. Same
shape DVSL and LBDC use. The `"2b"`/`"3b"` aliases work (line
167-177 maps to `doubles`/`triples`), but `"avg"` is silently dropped.

### C5. Player creation via `api/captain-add-player` and `api/captain-roster` writes `status: "active"`, but provision script does NOT

This is the other half of C1. Lines 153, 177 of `captain-add-player`
and `captain-roster` correctly stamp `status: "active"` on new docs.
But the bulk-CSV provision flow (`scripts/provision.ts:393-405`)
omits it. Any future tenant provisioned via the CSV path inherits
the same empty-roster bug as SFBL. The patch script
`scripts/patch-lbdc-stat-columns.ts` model is the right shape for
fixing this on already-provisioned tenants; the canonical fix is to
add `status: "active"` to the provision write.

## 2 — High (visible UX bugs / data quality / missing values)

### H1. `new Date(game.date)` TZ trap is everywhere, not just the captain ScheduleTab

The fix for the "every game at 8 PM" bug (commit `1044d99` and
others) was applied to ScheduleEditor + ScheduleTab + format-time
but ~15 other surfaces still parse date-only strings directly:

- `app/captain/page.tsx:1086, 1250, 1257, 1311, 1340` — NextGame
  spotlight + game-day hero + game-row meta
- `components/captain/AttendanceTab.tsx:314, 809, 816` — RSVP date
  labels + remind-waiting push body
- `components/profile/PlayerAvailabilityPanel.tsx:274, 281` — public
  RSVP panel date/time
- `components/BoxScoreContent.tsx:149, 322, 329, 420, 429` — game
  hero, preview blurb, completed-game meta line
- `components/ui/Ticker.tsx:207` — status label fallback
- `app/teams/[teamId]/page.tsx:1053` — recent-games rail date
- `lib/stats/recap.ts:178` — recap headline date

For SFBL these mostly read combined-ISO `date` values and look correct
in EDT/EST. For LBDC they all skew one day. The systemic fix is one
helper — e.g. `formatGameDate(g: { date, time })` — and a sweep.
Until then, every page that touches a game date is showing wrong dates
for LBDC users not in UTC.

### H2. `captain-payment` accepts admin without a `teamId` and lets them write any player's payment doc

`app/api/captain-payment/route.ts:80-90` sets `isAdmin=true` and
proceeds without forcing the admin to pass `teamId`. Line 102-107
only blocks captains whose `team_id !== captainTeamId`. So an admin
token with a typo'd `playerId` body silently writes a payment row
for the wrong player. Not a security hole (admin can do that anyway)
but it's the only `captain-*` endpoint without the
"admin must include `teamId`" guard that `captain-add-player:84-90`
and `captain-roster:108-116` enforce. Consistency would catch
fat-finger bugs in admin tooling.

### H3. `api/player-link` exposes per-tenant player linkage to any signed-in user via Host header trust

`app/api/player-link/route.ts:79-95` lets the caller-host check pass
if the user has ANY claim for the requested leagueId, OR if the
request's `Host` resolves to that tenant. The second branch is
correct as defense-in-depth, but it relies on `resolveTenant()`
serving the same hostname → tenant mapping the middleware uses. If
a future `LEAGUEENGINE_HOST_ALIASES` env var goes wrong (or
Edge Config gets stale), a request from `sfbl.leagueengine.com` with
`leagueId: "lbdc-staging"` in the body could still pass when the
caller has no claim, and read every email under
`/leagues/lbdc-staging/players/*/_private/contact`. The leak is bounded
to enumeration-by-email (one match per request), but it's the only
cross-tenant write path under any kind of self-service control. Worth
a follow-up unit test: `resolveTenant(parseHost("sfbl.leagueengine.com"))`
must return `{id: "sfbl"}` even when env vars are missing.

### H4. `api/admin-grant-claim` only matches `[a-z0-9_-]` for playerId; real LBDC player IDs may include uppercase or dots

`app/api/admin-grant-claim/route.ts:130` uses `TEAM_ID_RE =
/^[a-z0-9_-]+$/`. LBDC players have IDs like `bryce-bickel-12` (lowercase
slug — matches), but `transform-lbdc.ts:308` allows `status: "active" |
"dl" | "released" | "unknown"` and IDs are stringified numeric primary
keys ("1", "2", "157"). Pure numbers pass the regex. Edge: if any
LBDC player ID slipped through with uppercase characters, role:
"player" granting fails with a 400 that's confusing for the admin.
Low risk; flagging because the regex is silently restrictive.

### H5. Box-score `meta` row date shifts one day for LBDC on the public page

`components/BoxScoreContent.tsx:149` — `new Date(date)` for the meta
row "🗓 May 16" rendered below the FINAL hero. For LBDC games (date-
only), Friday-night EDT users see "May 15" instead of "May 16". Same
root cause as H1, but particularly visible: this is the public-facing
box score that shows up in shared iMessage links.

### H6. `seed-lbdc-to-firestore.ts` and live `/leagues/lbdc-staging` doc may drift

Couldn't run live inspection (sandboxed). Code-only comparison
between `scripts/seed-lbdc-to-firestore.ts:269-348` and the patch
scripts in `scripts/patch-lbdc-*.ts`:

- `seed-lbdc-to-firestore.ts` writes `captain.passwordless: true` and
  `admin.passwordless: true` but NOT `admin.password`. The password
  is set out-of-band by `patch-lbdc-admin-password.ts`. A re-run of
  `seed-lbdc-to-firestore.ts --provision` would leave the existing
  `admin.password` intact (set merge:true), so this is safe — but
  surprising for whoever inherits the script.

- `seed-lbdc-to-firestore.ts:281-285` writes the expanded
  `stat_columns` array (HBP/SF/SAC/FC/ROE/CS already in there). The
  `patch-lbdc-stat-columns.ts:44-61` writes the same list. Currently
  in sync; if you edit one, edit both or delete the patch script.

- `seed-lbdc-to-firestore.ts:318-320` writes `nav.hide:["News","Team
  Registration","Team Waiver","Store"]`. The `patch-lbdc-nav-hide.ts`
  file isn't open, but the seed list is the canonical place. Worth
  a future cleanup pass: collapse the half-dozen one-shot patch
  scripts into the seed and delete them.

### H7. `app/print/roster/[teamId]/page.tsx:53` uses only `active !== false`, surfaces LBDC orphan players

Print-roster page filters with `d.data().active !== false` only —
LBDC's 1100 orphan docs (`status: "unknown"`, no `active` field)
pass through. Admin clicks "📄 Contacts (confidential)" expecting
their 15-person team and gets a 154-row list with orphans interleaved.

Fix: apply the same three-condition filter used everywhere else.

### H8. `app/sitemap.xml/route.ts:78-87` only excludes `active === false` and `walk_on === true`, surfaces orphan players to Google

LBDC orphans (~1100 docs) have neither `active === false` nor
`walk_on === true`, so they all end up in the sitemap. Google may
index thousands of player profile pages with "Player not found" or
stub data. Same fix as H7.

### H9. `lib/gcal.ts:144-150` mixes UTC and local-TZ math for non-allday events

The branch that builds a 2-hour calendar block constructs
`startDate = new Date(startIso + "-00:00")` (UTC interpretation),
then derives `endIso` via `getFullYear/getMonth/getHours/getMinutes`
(local TZ). The output `start.dateTime` and `end.dateTime` are
emitted as naive strings with a `timeZone` field, so the calendar
provider re-anchors them. If the server runs in UTC the math works;
if Vercel ever places the function in a non-UTC region the start /
end will disagree by the offset. Best fix: use only UTC throughout
(`getUTCHours`, etc.) since the strings then represent UTC unanchored.

### H10. `app/api/captain-schedule` admin can still pass any `teamId` but new game creation is gated correctly

Looking at lines 115-123: `!isAdmin && captainTeamId !== awayId &&
captainTeamId !== homeId` — admin bypasses the participant check
intentionally. That's fine. But there's no validation that
`game.date` is a clean date-only string before storage — line 162
just slices to 10 chars. If a captain POSTs `{date: ""}` for time
TBD, the doc gets `date: ""` and downstream pages will show "Invalid
Date" rather than "TBD". Low-impact polish.

### H11. `app/captain/box-score/page.tsx` `ColKind` and ALL_COLS index has `"avg"` not present, but SFBL config asks for it

See C4. When SFBL captains open the box-score editor today, they see
only AB/H/2B/3B/HR/RBI/BB/K — eight columns. They cannot enter R or
SB. Existing SFBL captains who tested this in pre-launch may have
already filed bug reports. Worth checking the `errors-log` collection.

## 3 — Medium (works but worth knowing)

### M1. `api/public-captain-claim` synthetic uid lets all visitors share one auth identity per team

`app/api/public-captain-claim/route.ts:183` —
`uid = "public-captain:<league>:<team>"`. Every visitor who signs in
for the same team writes to the same Firebase Auth user. Downstream
audit trails (`who edited this box score`) can only resolve back to
"a public-captain visitor for team X" — they can't distinguish two
captains of the same team submitting different lineups. Audit log
entry includes IP (`route.ts:198-199`), which is what you'd actually
correlate on, but acknowledge that `created_by_uid` / `updated_by_uid`
on box_scores and games is shared across all LBDC visitors. Document
this if you ever need to dispute a captain edit.

### M2. `api/public-captain-claim` legacy single-input mode (rawPassword without teamId) walks every team

`route.ts:157-170` — if only `teamPassword` is sent, the endpoint
fetches every team in the league and normalize-compares every
candidate string. For a 30-team league this is fine; at 100+ teams
it's a noticeable amount of work per claim attempt, and the
RATE_LIMIT of 60/IP/10min combined with this branch is one of the few
places where a determined caller could enumerate normalized team
strings (300 reads × few-byte responses). Defensible because every
candidate is already public-readable, but worth narrowing — the
two-step picker is what the LBDC UI uses; the legacy single-field is
admin tooling only and could require an explicit `--legacy` flag.

### M3. Captain edit-game time validation is regex-only; "9:5" passes `/^\d{1,2}:\d{2}$/` is wrong, but `/^\d{1,2}:\d{2}$/` actually requires 2 digit min

Re-read: `/^\d{1,2}:\d{2}$/` requires exactly 2 digits in the
minutes. "9:5" doesn't match (would need "9:05"). So OK. But "25:00"
DOES match. A captain who types "25:00" gets it stored, and `formatTime12`
returns the original string (`format-time.ts:46`). Bound the hour to
0-23 in both the API and the formatter — that's the right place for
defense in depth.

### M4. `lib/gcal.ts` writes Google Calendar events but no consumer reads them back; admin-gcal route should idempotency-check

This is more of a "the integration is wired but I haven't seen it
exercised in the recent commits." Worth confirming the SFBL
preview tenant doesn't have stale `gcal_event_id`s from an earlier
sync that point at deleted events.

### M5. `app/api/check-pending-nav` reads `/pending_nav` (top-level), filters by `leagueId` in-memory

Line 53-57: `db.collection("pending_nav").where("auth_uid", "==", ...)
.where("leagueId", "==", ...).get()`. The two `where`s together
require a composite index that has to be added in `firestore.indexes.json`.
If it's missing, the query 500s. Check whether the index is
declared.

### M6. `recalcLeague` is invoked on every captain submit and reads the entire `/box_scores` collection

`app/api/captain-submit/route.ts:435` → `recalcLeague(db, leagueId)`
reads `leagues/<id>/box_scores` and every `leagues/<id>/players/<id>`
referenced. At LBDC's scale (1100 players, 200+ box scores), each
captain submit is a few hundred reads + ~50 writes. Cost-OK for now;
expensive at 5+ tenants. Mark for the standings-cloud-function v2
work in PLAN.md §10.

### M7. SFBL hardcoded config has no `flags` block

`lib/tenants.ts:193-219` doesn't declare `flags`. The audit prompt
mentioned `flags.captain_passwordless` as the gate name — that flag
doesn't exist anywhere in the code. The actual gates are
`captain.passwordless` (line 33 of tenants.ts) and `admin.passwordless`
(line 37). SFBL config doesn't set either, so both code paths fall
through to magic-link. Documenting because the audit brief assumed
the wrong gate name.

### M8. `scripts/seed-lbdc-to-firestore.ts` re-run with `--provision` overwrites `migrated_at`

Line 346: `migrated_at: new Date().toISOString()`. With `merge:true`
this updates every re-run. Not catastrophic; just a "this field is
not what you think it is" surprise for whoever queries it later.

### M9. `formatTime12` returns the input unchanged for already-formatted strings

`lib/format-time.ts:46` — defensive pass-through. Good. But it also
returns the input for invalid HH (`25:00` per M3, also negative values
the regex doesn't catch). The function name suggests it always
returns a 12-hour string; the contract is actually "returns 12h if
parseable, else echoes." Worth a JSDoc note.

### M10. `app/api/public-admin-claim` constant-time compare pads to 256 chars

`route.ts:48-55` — works for typical admin passwords (~10-30 chars).
But a password longer than 256 gets silently truncated and matched
on the truncation. Real risk = zero (nobody sets a 257-char
password), but the failure mode is "passwords longer than 256
appear to match arbitrary 256-char strings", which a security
auditor will flag.

### M11. `app/captain/page.tsx:1311-1318` `isGameToday` uses local TZ

The "is today" check parses `game.date` with `new Date()` and reads
`getFullYear/getMonth/getDate`. For LBDC date-only strings, this is
local-TZ-shifted and may consider a Sat 12 PM EDT game "yesterday"
during the early-morning hours of game day in a different timezone.

### M12. `lib/site-data.ts:165` caches by tenantId but uses module-level Map

Process-local in-memory. Per-instance only. On Vercel that's fine
across page navigations on warm functions, but every cold-start
re-reads the full `/games` + `/teams` collection. With 30s TTL on
short bursts that's typically OK. Worth flagging because the comment
explicitly calls out the assumption.

### M13. `app/api/admin-bulk-invite` magic links use `continueUrl=/login/finish` by default

`route.ts:121` — fine for SFBL. For LBDC tenants under
passwordless mode, admins don't typically use bulk-invite (they share
the team password instead). If they do, the magic link still works
but is a redundant path. Cosmetic.

### M14. `lib/notifications/send.ts:204` writes pending_nav using `db.collection("pending_nav").doc()`

The collection is top-level, not under `leagues/<id>/`. Each doc has
a `leagueId` field. Cross-tenant isolation depends on the filter at
the read site (`check-pending-nav/route.ts:55-57` AND the firestore
rule at `firestore.rules:371-375` which restricts read to
`request.auth.uid === resource.data.auth_uid`). The combination is
safe, but the leagueId is in a field, not the path, which is the
opposite of the platform's everywhere-else convention. Already
documented in `firestore.rules:362-368`; flagging for future
consistency.

### M15. SFBL hardcoded config in `lib/tenants.ts:181-182` ships the Firebase API key split into halves

`SFBL_FIREBASE_API_KEY = "AIzaSyBTG3b_rFv" + "D6s-KLvdi5GHIRtQLVaRuUf4"`.
GitGuardian-evasion trick. Fine in concept (the key is public anyway,
NEXT_PUBLIC_*), but make sure the actual Vercel env vars
(`NEXT_PUBLIC_FIREBASE_API_KEY`) hold the same value — otherwise
SFBL preview uses the hardcoded one and live SFBL uses the env one,
and they could drift.

### M16. `components/captain/QuickScoreTab.tsx:84-85` appends `T12:00:00` to date string blindly

Works when `a.date` is a clean YYYY-MM-DD. Produces an invalid Date
when `a.date` is already a combined ISO ("2026-05-16T19:30:00.000ZT12:00:00"
parses as Invalid Date). Sort logic falls through to NaN compare =
0, so the rows show in stable but undefined order. Use combineDateTime.

### M17. `app/api/captain-schedule` create flow lets captain set status freely on creation

Line 169: `status: "scheduled" as const`. Hard-coded — captains can't
create a game already in "final". Good. Edit flow correctly restricts
to ALLOWED_STATUS. But the edit path doesn't write `created_by_uid`,
so audit-trail "who created this game" is only on creates. Not a bug,
just a documentation note.

### M18. `app/api/page-content` validation only on action presence, no rate limit

`page-content/route.ts` (read earlier; admin-only via claim). Same
posture as other admin endpoints — relies on the admin claim being
hard to get. With LBDC's shared admin password, "admin claim" is
"anyone with the password." Combined with the 20-attempt/10min rate
limit on `public-admin-claim`, this is the v1 risk surface. Worth
considering whether admin pages should also rate-limit writes.
